I want to build a agentic system. It will be split into different parts for different things and be completely event driven. Here are the parts i am thinking about

- Core: The core will be the main control hub, this is where all agents are running, it will house the event system, memory, it will be the rest api for interacting with the entire system and so on. When when some events are fired it will be incharge of running different actions depending on the event which ill explain more in a few.

- Node: This is where actions take place, the core will have a node automatically but node allow users to add other machines their agent can work on in a secure way. For example I may have a "home-server" and "dev-machine" node my agent can work on. These will have to have a secure way of connecting where the node shows a code or something when starting, then the core cli will allow us to link it with that code. Id prefer if we could do this in a way that neither machine had to have exposed ports, because some users may wish to have nodes on their external services and messing with public facing ports is inheritly annoying or insecure. Nodes that have been authenticated once should not have to be authenticated again and should persist whatever token we come up with in ~/.auxlry/store/token.

- Interface: This is the llm the user communicates with. It should never be blocked by longer running tasks. The whole purpose of the interface is to reply or delegate. Delegating here is through two other agents in the system Synapses, and Operators more on them later. But when the interface recieves a incomming message event from an `Adapter` it will reply, skip or delegate. Its aware of whats going on in the system so it knows what nodes are active, and what synapses and operators are active.

- Adapters: These are a unified logic layer to send and recieve messages on platforms like discord though serentity. Or others like telegram or webhooks. They emit events the interface they're attached to can reply too and then listen for events from the interface to interact with the specific platform for example, reply, typing indicator, edit, and so on. Each configured interface gets one adapter. One interface llm per adapter keeps that interface focused where it should. But the config should allow multiple of the same adapter for example if I want one interface for all my public discord channels I should be able to do that, and have one for my dms, and one for the admin channels.

- Synapses: These are the thinkers, when the interface needs to think, plan, or save or remember from memory this is what it uses. This is a seperate llm that can spend the time it needs to fully get the info it needs, then can ask for workers to do jobs if applicable. While the interface can spawn operators for super quick tasks, the ideal route is synapse -> operators. Synapses when complete will emit an event that the interface that spawned it can then elaborate its reply with.

- Operators: These are the actors. They have similar tools to Claude Code, they can do things on nodes. While they run in the core their actions happen on nodes. By default actions will happen on the built in core node, but they can if asked specifically by name work on other nodes connected to the core. They will be sandboxed to ~/.auxlry/workspace/\* by default but in the config can be set to `system` to work through the home directory of the system they are on.

- Memory: This is the most important part of the set up but needs further planning, this has to be fast and completely local with all memory storage files in ~/.auxlry/store/\*. Been looking at https://github.com/agentralabs/agentic-memory for a possible candidate so please set a sub agent to research that and rust based alternatives. Or we can roll our own with fastembed and lancedb if none of those work.

- Storage: We will use a sqlite database for storing all events, messags, and so on needed to build out a dashboard. While this wont be the main memory storage/access we want it to persist them still in a way that easy to query and edit for the more agent centric memory set up we plan out. Think of this like the display/trace/human needed data to know whats going on in the system while the memory is for the agent.

The config will be placed in ~/.auxlry/config.yml on linux and the same standard best practice locations on other systems. The binary will mainly be a cli that starts/stops/restarts the daemon process. Core and Node will both be contained in the same binary but prefixed with `core` or `node` like auxlry core start or axlry node restart. The ai framework we will be using is rig so please have a sub agent research the latest version of that https://github.com/0xPlaygrounds/rig. Finally each llm in the system should be able to have a different model. In the config we should be able to customize the synpases model, the operators model, and the interfaces model. For the rest api we want to use the best practices in rest api development with openapi specs, and for the project overall we want to have the rust best practices so its easy to pick up read, fork, contribute and use. We will use a .md.j2 templateing setup for all agent prompts in /prompts/en/\* to start so multilingual support can be added later when desired. Config should default to `en` all adapters will get a little prompt so the agent knows that its adapter requirements are for formatting. For example discord can support full markdown but has a size cap, other adapters will differ so we need some way the interface to undertand that for each one
